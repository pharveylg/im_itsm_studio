# Documentation Style Guide (DRAFT)

> **Status:** DRAFT — seeded placeholder, to be ratified under OAS-000 governance.
> This guide governs all OAS standards and Knowledge Base content.

## Purpose

Define the voice, structure, and quality expectations for OAS documentation so that every
standard reads as a single, cohesive library.

## Voice and Tone

- Write in a clear, imperative, professional register.
- Use normative language consistently: **Shall** (mandatory), **Should** (strong
  recommendation), **May** (optional), **Must Not** (prohibited).
- Address the reader as "the analyst" or "the operator"; avoid ambiguous first-person.

## Structure

Every standard follows the Standard Document Structure defined in OAS-000 / the library
structure:

1. YAML Metadata
2. Purpose
3. Scope
4. Guiding Principles
5. Inputs
6. Required Evidence
7. Methodology / Standard
8. Findings / Assessment
9. Recommendations
10. Lessons Learned (where applicable)
11. Quality Assurance Checklist
12. AI Operating Standard
13. Related Standards
14. Related Knowledge Base
15. Revision History
16. Future Revision Register

## Quality Bar

- Evidence-based, internally consistent, cross-referenced, version-controlled.
- Must not contradict OAS-000.
